# 一、生产级多-agent架构

## 1）总架构

```text
Client Request
    │
    ▼
API Controller
    │
    ▼
Orchestrator / Scheduler
    ├── Prompt Registry（从文件加载提示词）
    ├── Task State Store（任务状态）
    ├── Attempt Store（每轮代码、日志、issue、修复记录）
    ├── Notice Library（已验证有效的注意事项）
    │
    ├── Agent A: Request Planner
    ├── Agent B: Manim Code Expert
    ├── Tool 1 : Render Executor
    ├── Agent C: Error Diagnoser
    ├── Agent D: Repair Agent
    ├── Agent E: Fix Validator / Learning Agent
    └── Tool 2 : OSS Uploader
```

核心原则：

* **LLM 只负责认知型工作**：理解需求、写代码、诊断错误、归纳经验
* **程序模块负责确定性工作**：执行渲染、判断成功、比对 issue、上传 OSS、落盘存档

这条一定别反。

# 二、职责拆分

## Orchestrator / Scheduler

这是主脑，不是普通 agent。

职责：

* 接收请求
* 校验参数
* 从文件读取 prompt
* 组织上下文
* 调度每个 agent
* 控制重试次数
* 记录 attempt 历史
* 控制“什么时候允许写入注意事项库”
* 生成最终响应

它不直接写代码，只管流程和状态。

---

## Agent A：Request Planner

职责：

* 解析 `projectBrief`
* 解析 `timedScenes`
* 生成统一中间表示 IR
* 检查潜在风险点
* 给代码专家提供稳定输入

输出：

* `normalized_request.json`
* `scene_ir.json`
* `risk_report.json`

### 为什么必须有这个 agent

因为直接从原始 JSON 生代码很脆。
你要先把输入整理成“能执行的场景计划”。

---

## Agent B：Manim Code Expert

职责：

* 根据 `scene_ir.json` 生成完整 Manim 代码
* 参考提示词文件、注意事项库、官方规范
* 产出结构化、可修复、可维护的代码

要求：

* 尽量模块化
* 每个 scene 单独封装
* 字幕、样式、时间轴、对象构建分离
* 尽量少写魔法常量
* 不要把所有逻辑塞进一个 `construct()`

Manim 官方文档说明，`Scene` 是基本画布，通常通过重写 `construct()` 来组织内容，并用 `add()`、`play()` 来管理对象与动画；CLI 也支持用 `--output_file`、`--media_dir`、`--log_dir`、`-r`、`--fps` 控制渲染输出，因此代码层和执行层最好围绕这些稳定接口设计。 ([Manim Community | Documentation][1])

---

## Tool 1：Render Executor

这个不要用 agent，写成程序模块。

职责：

* 写临时代码文件
* 调 Manim CLI
* 收集 stdout / stderr / exit_code
* 检查视频文件是否生成
* 生成渲染报告

### 成功判断标准

必须同时满足：

* `exit_code == 0`
* 视频文件存在
* 文件大小 > 最小阈值
* 能读取视频元信息
* 元信息和预期匹配度在可接受范围内

不要只看“没报错”。

---

## Agent C：Error Diagnoser

职责：

* 解析控制台日志
* 抽取 issue
* 给出 issue 指纹
* 标注问题阶段和根因候选

输出示例：

```json
{
  "issues": [
    {
      "issueId": "ISSUE_001",
      "stage": "runtime",
      "errorType": "ValueError",
      "rootCauseLabel": "invalid_run_time",
      "normalizedMessage": "run_time <= 0",
      "confidence": 0.94,
      "evidenceLines": [
        "Scene.play() has a run_time of 0 <= 0 seconds"
      ]
    }
  ]
}
```

---

## Agent D：Repair Agent

职责：

* 针对指定 issue 修代码
* 不允许无目的大改
* 输出“修复目标 + 修改理由 + 代码补丁”

输出示例：

```json
{
  "targetIssueIds": ["ISSUE_001"],
  "fixStrategy": "Clamp all animation durations to at least 1/fps",
  "expectedOutcome": "Remove invalid zero-duration animations",
  "patchSummary": [
    "Added safe_duration() helper",
    "Wrapped play/wait durations before execution"
  ]
}
```

---

## Agent E：Fix Validator / Learning Agent

这是最关键的一个。

职责：

* 比较修复前后 issue 集合
* 判断某个修复方案是否对目标 issue 生效
* 决定是否允许写入注意事项库

### 它判断的是“问题是否解决”

不是“整次运行是否成功”。

---

## Tool 2：OSS Uploader

也是程序模块，不要交给 agent。

职责：

* 上传最终视频
* 返回 OSS 链接
* 记录 object key、etag、bucket、region

阿里云 OSS 官方文档里，Python SDK 支持 `put_object` 做简单上传；V2 文档说明单对象简单上传最高 5 GiB，并默认做 CRC-64 校验。生产环境更建议用 STS 临时凭证，而不是在服务里长期暴露固定 AccessKey。 ([Alibaba Cloud][2])

# 五、Fix Validator 的判定规则

这个必须写死，别让模型自由发挥。

## 定义三种结果

### 1. 有效

满足全部条件：

* 修复方案声明了目标 issue
* 下一轮运行里，该 issue 指纹消失
* 或同根因错误不再出现
* 没有证据表明它只是被更早的新错误遮住
* 最好还能看到流程推进到更后阶段

### 2. 无效

满足任一条件：

* 目标 issue 仍存在
* 文案变了，但根因等价
* 修复没有触达目标代码路径
* 修复方式和 issue 无关

### 3. 暂不判定

满足：

* 引入了新的更早阶段阻塞错误
* 导致原目标问题根本没机会被运行到

这类不要学习，也不要算有效。

---

## issue 对比算法

先做 issue 标准化，再比较：

```python
def is_same_issue(old_issue, new_issue):
    return (
        old_issue.stage == new_issue.stage
        and old_issue.error_type == new_issue.error_type
        and old_issue.root_cause_label == new_issue.root_cause_label
        and similarity(old_issue.normalized_message, new_issue.normalized_message) > 0.82
    )
```

再做修复验证：

```python
def validate_fix(target_issues, new_issues, introduced_earlier_blocker: bool):
    if introduced_earlier_blocker:
        return "blocked"

    unresolved = []
    for target in target_issues:
        if any(is_same_issue(target, ni) for ni in new_issues):
            unresolved.append(target.issue_id)

    if unresolved:
        return "ineffective"

    return "effective"
```

# 六、注意事项库的写入规则

只允许写入**已验证有效**的经验。

## 推荐格式

```yaml
- id: NOTICE_0012
  issue_type: invalid_run_time
  condition:
    stage: runtime
    error_type: ValueError
    contains: "run_time <= 0"
  rule: "所有动画和等待时长必须 > 0，且建议 clamp 到 max(value, 1/fps)"
  evidence:
    verified_attempts: 4
    success_rate: 1.0
  status: validated
```

## 不要直接把自然语言废话写进去

例如这种就很差：

```yaml
- rule: "以后写代码要小心一点，不要报错"
```

这是垃圾。

# 八、调度器主流程伪代码

```python
def process_render_task(request: dict) -> dict:
    task = create_task(request)

    prompts = load_prompts()
    notices = load_validated_notices()

    normalized = planner_agent.run(
        request=request,
        prompt=prompts["planner"]
    )
    save_json(task.path / "normalized_request.json", normalized)

    current_code = code_expert_agent.run(
        scene_ir=normalized["sceneIR"],
        notices=notices,
        prompt=prompts["code_expert"]
    )

    for attempt_no in range(1, task.max_attempts + 1):
        attempt_dir = prepare_attempt_dir(task, attempt_no)
        save_text(attempt_dir / "generated.py", current_code)

        render_report = render_executor.run(
            code_file=attempt_dir / "generated.py",
            output_name=f"{task.task_id}.mp4"
        )
        save_json(attempt_dir / "render_report.json", render_report)

        if render_report["success"]:
            video_path = render_report["video_path"]

            oss_result = oss_uploader.upload(video_path)
            final_result = {
                "success": True,
                "videoUrl": oss_result["url"],
                "taskId": task.task_id
            }
            save_json(task.final_dir / "final_result.json", final_result)
            return final_result

        issues = diagnoser_agent.run(
            code=current_code,
            render_report=render_report,
            prompt=prompts["diagnoser"]
        )
        save_json(attempt_dir / "issues.json", issues)

        repair_result = repair_agent.run(
            code=current_code,
            issues=issues,
            notices=notices,
            prompt=prompts["repair"]
        )
        save_json(attempt_dir / "repair_decision.json", repair_result["metadata"])

        new_code = repair_result["code"]

        validation = validator_agent.run(
            previous_target_issues=repair_result["metadata"]["targetIssueIds"],
            previous_issues=issues,
            next_code=new_code,
            prompt=prompts["validator"]
        )

        if validation["shouldLearn"]:
            append_notice(validation["candidateNotice"])

        current_code = new_code

    return {
        "success": False,
        "taskId": task.task_id,
        "message": "Render failed after max attempts"
    }
```

# 九、你这个系统还该补的两个模块

## 1）静态检查器

在真正渲染前，先做一次 cheap check：

* Python 语法检查
* scene class 是否存在
* 明显空引用
* duration 是否非法
* 外部资源 URL 是否为空
* subtitle 时间区间是否合法

这样能省很多轮。

---

## 2）视频质检器

渲染成功不代表视频合格。
建议再加一层：

* 时长是否接近 scene 总时长
* 画面是否全黑
* 是否有空白超长停留
* 是否有字幕出框
* 是否有严重遮挡

这层可以先做成规则引擎，后面再加视觉模型。

# 十、我给你的最终建议

你这个项目，最容易做歪的地方有两个：

## 第一，不要把“工具执行”交给 agent

渲染、上传、文件检查、issue 对比，必须程序化。

## 第二，不要把“经验沉淀”写成 prompt 污染

经验必须结构化、可验证、可撤销、可统计。
